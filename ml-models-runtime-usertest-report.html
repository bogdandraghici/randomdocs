01Executive summary

Same feature, three configurators, remarkably consistent verdict.

Verdict: The runtime ML-models experience is fundamentally sound and well-received — all three testers navigated it largely unaided, built accurate mental models of the review → correct → corpus → retrain loop, and rated the tooling useful and realistic. The friction is concentrated in discoverability and copy, not structure: users reach the right screens but can't always tell why an item is there, what a number means, or that a control is theirs to change. Two genuine design tensions (threshold ownership, LLM role) need a product decision rather than a copy tweak.

What's working

  • Runtime placement matches everyone's instinct — none looked in Config.
  • The "candidates" → "models" rename was a unanimous relief; the old label had actively confused testers.
  • The correct → corpus → retrain loop is self-explaining.
  • The LLM daily budget cap read as a genuine safety feature.

What's hurting

  • Review-queue entry criteria are buried — the single most-confusing thing for one tester.
  • Batch names feel system-owned and go unnoticed, yet they're the key to slicing data.
  • What the LLM actually does (pre-classify vs. judge) is ambiguous — and drives cost fear.
  • Unlabeled numbers ("100%", "used by 4 builds") leave users guessing.

  • Role coverage is narrow. All three testers were configurators. No end-user and no client-annotator was tested — yet the human-in-the-loop debate (who corrects near-identical documents) hinges precisely on those roles. Treat human-in-the-loop findings as hypotheses, not conclusions.
  • Prototype artifacts muddied some findings. "Open in run logs" was broken, and the review-list count/select control behaved oddly — both drew comments that may not reflect the intended design. Re-validate on a fixed build.
  • Language. Sessions were Romanian-primary with English technical terms; quotes below are verbatim Romanian with an English gloss.
  • Facilitation. Several correct readings came only after a prompt ("scroll and compare the LLM vs. ML rows"). Where a finding depended on a nudge, it's noted — unaided comprehension is the higher bar.

04Insights by theme

The eight topic clusters that recurred across sessions, synthesized.

Navigation & discoverability

Runtime vs Configlabelslogs

All three reached ML Models through Runtime, confirming the placement — though Delia's first instinct was "data sources" before she corrected. The section label "runtime content" itself carries no meaning (George oriented by contents, not name). Users expect a runtime log of every tested document and its classification (George), and expect the "view" control to open the workflow config — because that's the behaviour elsewhere — where it actually shows logs (Delia). Returning to a model's detail without triggering a retrain wasn't obvious (Cătălin).

Naming & labels

candidates→modelsimportversion100%

The "candidates" → "models" rename is the standout copy win. Remaining snags: the naming hierarchy — an "ML Model" that opens into many inner "models" — reads oddly; Cătălin would call the inner items "versions." A bare "100%" is assumed to be accuracy but isn't labeled. The "critical" class tag is read as an informative degradation warning, but its explanatory copy contradicts itself ("can't be promoted to main," on a model already main). "Import" was misread as "export" by one tester.

Review queue & entry criteria

criteria opacityscopecorner cases

The criteria that decide which pages land in the queue were the single most-confusing thing (Delia looked in settings and still couldn't state the rule). Once opened, the settings themselves read correctly. Users built accurate scope models: the queue runs on the whole classifier, not one model (George, self-corrected), and holds corner cases from real client data surfacing after a 100%-in-dev deploy (Cătălin). The fix is surfacing the criteria at the queue, not rewriting them.

Confidence & thresholds

<60%top-2 closeownership tension

Both triggers — score below 60%, and top-two classes within ~10 points — were understood, often from a good worked example (residence permit 73% vs. driver's license 70%). The open design question is where the threshold lives: George argues classifier-level ("you can't be only somewhat sure a doc is an ID"); Cătălin wants it exposed as a workflow gateway so the process branches and never blocks. 1 finding flagged for follow-up validation

LLM assist & budget

pre-classify vs judgedaily captrust curve

The intended behaviour — the LLM pre-classifies queued items blindly (ignoring the ML guess), capped by a daily budget — landed for George and (after a prompt) Delia. Cătălin instead inferred an LLM-as-judge on every call, which fuelled real worries: cost (the ML model exists to cut AI spend), ~20s latency, and false overrides of correct predictions. George praised the budget cap as "a lot of safety." George also described a natural trust curve — verify everything manually first, lean on the LLM as it proves accurate.

Corpus & retraining

batch columnselect only newsuggestions

Reviewed pages feeding the corpus for retraining is well understood and valued ("more variety, closer to production reality"). But isolating just the new / just-reviewed items is hard: there's no "select only new" button, users fall back to filtering by the batch column — which none noticed unaided. Training suggestions aren't equally legible: struck-through "cut" suggestions read instantly, "+add" ones get missed (Cătălin saw 2 of 6). The "149/72 since last training" count also caught users unaware.

Batch names

perceived controlslicinginline validation

Every tester treated batch names as system-given defaults they couldn't control ("first added," "most common class," "review Q 13 iulie"). Once prompted, all saw the value: slicing data to include/exclude a set or inspect one document type later. Delia's concrete UX preference for a required batch name: an inline "required" field that turns red and blocks continuing (a modal "felt like more steps"), plus a delete "×" to signal the batch is hers to control.

Human-in-the-loop & end-user impact

who correctsend-users outpromote flow

Consensus: the review queue is invisible to and off-limits for end-users — it's the configurator's tool, reached after go-live. The nuance: Cătălin doubts a configurator can correct near-identical documents alone (on the VP project he "went with the guess") and wants the uploading user to label them. On promotion, George expects train → new version → manually select as main, with no build needed and — for himself — no perceived risk, matching FlowX's continuous-improvement story.

06Prioritized issue backlog

Ranked by severity × how many sessions surfaced it. Each with a recommended direction.

#IssueSeen byRecommended fix
P1 Review-queue entry criteria aren't discoverable — users can't state why items are there. Delia (top), George, Cătălin Surface the active criteria inline on the queue header (e.g. "showing pages <60% confidence, close-call classes, and LLM-flagged"), not only inside the settings modal.
P1 Batch names feel system-owned & go unnoticed, blocking the "isolate this batch / only-new" job. All 3 Make batch naming visibly editable (input + delete "×"), add a "select only new / this batch" action, and raise the batch column's prominence.
P1 LLM role is ambiguous (pre-classify vs. judge), driving cost/latency fear. Cătălin (fear), Delia (slow) State plainly that the LLM pre-classifies queued items, blind to the ML guess, within a daily cap — it does not judge every live call. Reinforce near the "LLM assist" tag.
P2 "class" vs "model prediction" columns — unclear which is truth vs. prediction; "reviewed" state unsignalled. Delia, Cătălin Rename to make truth editable / prediction read-only; add a "reviewed class" label & distinct row background; never mark reviewed by default.
P2 Naming hierarchy: an "ML Model" that contains many "models." Cătălin Call the inner items "versions."

P2 "Import" misread as "export." Delia Directional icon + "Import trained model from another project" helper.
P2 Accuracy basis unclear — computed on UAT data or the original training set? George Label / tooltip stating exactly what the score is measured on.
P3 Training-suggestion legibility — "+add" suggestions get missed vs. struck-through "cut." Cătălin Give "+add" equal visual weight; show the suggestion count.
P3 "critical" tag copy contradicts itself ("can't be promoted to main" on a main model). Cătălin Rewrite to describe the degradation, not a false constraint.
P3 "view" opens logs, not workflow config — inconsistent with the rest of the product. Delia Align behaviour or relabel the control.
P3 Two "Active Policy" rows & weak "runtime content" label. Cătălin, George Explain the duplication; give the section a name that evokes its contents.

09Open questions & follow-ups

What this round couldn't settle.

  • Threshold ownership — classifier-level default vs. per-node override. A product decision, not a usability tweak. (George vs. Cătălin.)
  • Validate LLM copy unaided — one finding was explicitly flagged for follow-up: confirm users grasp the review-queue triggers without a facilitator's worked example.
  • Accuracy basis — decide and label what the score is computed on; George's UAT-only assumption is currently unverified.
  • Test the missing roles — run a session with an actual end-user / client-annotator before committing to who corrects near-identical documents.
  • Re-test on a fixed prototype — "open in run logs" and the review-list select control were broken; re-confirm affected findings.
